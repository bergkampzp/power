## 一、整体架构总览

项目建立了从 P0→P4 的层次化模型体系，覆盖实时电价(RT)和日前电价(DA)两个预测任务：

| 模型 | 目标 | 方法 | 粒度 |
|------|------|------|------|
| P0-v1/v2 | RT直接预测 | GBR + 时间特征 + 价格lag | 96点/24小时 |
| P0-DA | DA直接预测 | GBR (PostgreSQL) | 96点 |
| P1 | RT预测 + 特征丰富 | GBR/LGB/XGB 3模型ensemble | 24小时 |
| P2 | RT分解预测 | Savgol分解 + BiLSTM + 5模型ensemble | 24小时 |
| P3 | RT价差预测 | 预测spread(RT-DA) + 加回DA | 24小时 |
| P4 | RT混合自适应 | 异常度评分动态加权P2和P3 | 24小时 |
| P2+Weather | RT天气修正 | P2 + 物理模型天气修正系数 | 24小时 |
| P3-DA | DA价差预测 | 预测DA[D+1]-DA[D] + 3模型ensemble | 96点 |
| P4-DA | DA混合自适应 | 异常度评分动态加权P0和P3-DA | 96点 |

### 优点
1. **层次清晰**: P0(基线)→P1(全特征)→P2(分解+深度学习)→P3(价差)→P4(混合), 递进关系明确
2. **双任务覆盖**: 同时支持 RT预测 和 DA预测, 并采用了不同的数据源
3. **天气修正补充**: P2+Weather 通过物理模型引入气象因子, 思路合理

## 二、特征工程评审

### 2.1 特征总量统计
- P0-v1: ~30个特征
- P0-v2: ~55个特征 (增加季节编码)
- P1: ~70个特征 (+电网约束、预测值、断面信息)
- P2: ~80个特征 (+CEEMDAN分解分量)
- P2+Weather: ~90个特征 (+天气修正因子)
- P0-DA(PostgreSQL): 75个特征

### 2.2 特征分类与合理性评估

| 特征类别 | 数量 | 合理性 | 问题 |
|---------|------|--------|------|
| 时间编码 (hour_sin/cos/woy/doy/quarter等) | ~26个 | ✅ 必要 | 存在多重冗余: month_sin+cos与woy_sin+cos与doy_sin+cos信息重叠, 季度one-hot(q1-q4)与is_wet_season高度相关 |
| 价格历史 (lag/ma/std/momentum) | ~11个 | ✅ 必要 | lag_1/2/3/7 + ma_3/5/7 + std_3/7 + momentum存在共线性: price_momentum_3d=price_lag_1d-price_ma_3d, 线性组合 |
| 负荷/新能源/水电 lag | ~5个 | ✅ 必要 | 合理, 但只有1-2天lag, 中周期信息不足 |
| 供需缺口 (gap) | ~4个 | ✅ 核心 | gap = load - renew - hydro, 合理但连续差值意义有限 |
| 日前电价 (manwan_da/grid_da) | ~4个 | ✅ 合法特征 | DA在D-1出清, 是合法的前瞻特征, 应是最强信号之一 |
| 预测值 (renew_fc/load_fc/gen_fc/hydro_fc) | ~7个 | ⚠️ **需警惕** | 预测值为D日发布, 在walk-forward中D日预测值在D日可用——但"可用"的前提是D-1发布。需检查数据是否实际D-1可用 |
| 电网约束 (section/reserve/maint/channel) | ~15个 | ✅ 有意义 | 但冗余度高: reserve_up/reserve_down/reserve_ratio/reserve_change/reserve_up_lag1d中有5个相关特征 |
| CEEMDAN分解分量 | ~5个 | ⚠️ **潜在泄露** | trend/wave/spike是用全量数据(含未来)Savgol滤波计算, 在walk-forward中使用当日信息有泄露风险 |
| 天气修正 (solar_corr/wind_corr/cloud等) | ~15个 | ✅ 有意义 | 但weather_correction表有1179天数据(2023-01起), 而其他特征只覆盖~290天, join时可能产生大量NaN |

### 2.3 ⚠️ 关键问题: 多重共线性

**P2模型特征集(~80个)中存在系统性多重共线性**:

```
1. 时间编码群 (13个指标表示季节):
   month, month_sin, month_cos, week_of_year, woy_sin, woy_cos,
   day_of_year, doy_sin, doy_cos, is_wet_season, is_transition,
   q1, q2, q3, q4
   → 有效自由度约2-3, 15个特征表达2-3个独立周期

2. 价格统计群:
   price_ma_3d ≈ (price_lag_1d + price_lag_2d + price_lag_3d) / 3
   price_momentum_3d = price_lag_1d - price_ma_3d 线性组合
   price_ma_5d = (sum of lag_1~5) / 5 含lag_1~3的全部信息

3. 缺口群:
   gap_lag_1d = load_lag_1d - renew_lag_1d - hydro_lag_1d
   (用其3个分量再线性组合得到gap_change)
```

**对GBR/LGB/XGB的影响**: 树模型对共线性容忍度较高, 特征冗余不会导致过拟合, 但会:
- 稀释特征重要性分数 (真实信号被分散到多个相关特征)
- 增加不必要的训练开销 (特征数从30→80, 训练量2.7倍)
- 降低模型可解释性

**对BiLSTM的影响**: 80个特征输入LSTM, seq_len=7, 第一层就是 80→128(64×2), 参数约10K+。考虑到~290天×24h≈7000样本, seq_len窗口后约5000序列, 参数/样本比约2:1, 轻度过拟合风险。

### 2.4 ⚠️ 关键问题: DA预测中的特征泄漏风险

**da_price_model.py** 中从 `feat_da_features` 表加载特征, 特征工程由dbt完成。核心风险:

1. **`da_lag_1d` = DA[D-1] = 前一天DA出清价**, 但P3-DA中 spread = DA[D] - DA[D-1], 而后续通过 `da_baseline + spread_pred` 得到 DA[D] 预测值, 这个没问题。

2. **`rt_lag_1d` 和 `rt_da_spread_lag1`** 使用了实时价格滞后, 在DA预测场景下是合法的(D日RT在D日做DA预测时, 只能看到D-1的RT), 但需要dbt层确认时间戳对齐正确。

3. **`zero_yesterday`** 特征 (标记D-1零电价日), 需要确认dbt实现中没有混入当日信息。

## 三、模型选型评审

### 3.1 GBR (Sklearn) 作为基线

| 参数 | P0 | P1/P2 | 评价 |
|------|----|-------|------|
| n_estimators | 300 | 500 | 合理, 树越多拟合越强 |
| max_depth | 5 | 6 | 偏浅, 但对于~300天数据合适 |
| learning_rate | 0.05 | 0.04 | 标准范围 |
| subsample | 0.85 | 0.85 | 合理 |

**评价**: GBR选择合理。对于~200-300天×24h≈5000-7000样本, GBR 300-500棵树, max_depth 5-6, 参数/样本比合适。

### 3.2 LightGBM + XGBoost Ensemble

评估: **必要性存疑**

P1/P2中同时使用GBR+LGB+XGB 3个模型, 加上spread模型和LSTM=5模型ensemble。从代码输出的MAE看:

```
典型日: GBR=45.2  LGB=43.1  XGB=44.8  Spread=40.5  LSTM=52.3
Ensemble: 0.20*45.2 + 0.25*43.1 + 0.15*44.8 + 0.35*40.5 + 0.05*52.3
```

**问题**:
1. LSTM在该场景下效果最差(MAE最高), 权重却只有5%, 对ensemble几乎无贡献, 却增加了显著复杂度
2. GBR/LGB/XGB同为树模型, 相关性较高, ensemble的降方差效果有限
3. Spread模型权重最高(35%), 而DA_RT_spread本质上就是RT - DA, 如果DA预测偏差小, spread建模收益有限
4. 权重固定而非自适应, 不会根据各模型近期表现动态调整

**建议**: 简化到2模型ensemble (LGB + Spread), 或使用交叉验证自动调权重。

### 3.3 BiLSTM 评估

| 项目 | 值 | 评价 |
|------|-----|------|
| hidden_dim | 64 | 对于80维输入, 64合理 |
| layers | 2 | 标准配置 |
| dropout | 0.2 | 偏小, 建议0.3-0.5 |
| seq_len | 7 | 合理(一周周期) |
| batch_size | 32 | 合理 |
| epochs | 30-50 | 偏多(~290天数据, 50轮可能过拟合) |

**关键问题**: BiLSTM实际收益存疑

从P2代码输出的10天结果看, 每个测试日 LSTM 的 MAE 都是最高(最差):
```
20260313: LSTM=78.4 (其他35-45)
20260314: LSTM=82.1 (其他38-48)
...
```

BiLSTM在所有10个测试日都是最差的模型, 且差距显著 (MAE高50-100%)。BiLSTM增加了:
- PyTorch依赖
- GPU/CPU选择逻辑
- 约20倍于GBR的训练时间
- 序列数据构建的复杂性
- 训练不稳定风险

**结论**: BiLSTM在此场景下**不值得保留**。可能原因:
1. 电价序列非线性强、信噪比低, LSTM难以学到有效时序模式
2. 80维特征中大部分是截面信息, 纯时序信息(价格Lag)只占少部分
3. ~5000序列样本对于BiLSTM(参数~15K)偏少

### 3.4 Savgol滤波代替CEEMDAN (P2分解)

代码注释写CEEMDAN, 实际实现用 Savgol 滤波:
```python
# Trend: savgol_filter(price_signal, window_length=21, polyorder=2)
# Wave: savgol_filter(residual1, window_length=7, polyorder=1)
# Spike: residual1 - wave
```

**评价**:
- Savgol滤波是线性滤波器, CEEMDAN是自适应非线性分解。用Savgol代替CEEMDAN意味着失去信号自适应性。
- Savgol(21d, poly=2) 等价于 二次多项式局部拟合, 对于电价这种有突变尖峰的时间序列, 边界效应显著。
- **更严重的问题**: 分解在**全量数据**上执行, 然后用于walk-forward。Savgol的窗口长度为21天, 意味着第N天的分解结果使用了N±10天(前后各10天)的数据 → 在预测第N天时使用了未来10天的信息 → **data leakage**。

**建议修复**: 
- 改为滚动分解: 每天只使用到该天为止的数据计算分解
- 或标记Savgol为非因果滤波, 需要在walk-forward中重算
- 或直接移除分解特征 (从特征重要性看, trend/wave/spike的贡献度远低于价格lag)

### 3.5 分时粒度选择: 96点 vs 24小时

**现状**:
- P0-v1/v2: 96点 (15分钟粒度)
- P1/P2/P3/P4: 24小时 (小时粒度, 96→24聚合)

**评价**: 24小时粒度是合理选择。原因:
1. 负荷/新能源/水电数据本身就是小时粒度, 96点都是重复值
2. 日前电价也是96点, 但DA预测时价差建模在96点有意义
3. 目标变量(电价)15分钟波动小, 小时均值预测已经足够
4. 样本量从 96/天→24/天, 每个模型训练数据量减少, 但预测稳定性提升

**但**: P0-DA和P3-DA应该保持96点粒度, 因为日前电价在15分钟尺度上有实际交易意义。

## 四、训练/评估方法评审

### 4.1 Walk-Forward 设计

所有模型采用 walk-forward 回测:
```python
train_mask = df['date_key'] < test_date   # 只用过去数据
test_mask = df['date_key'] == test_date
```

**评价**: ✅ 设计正确。按天滑动时间窗口, 没有使用未来信息做训练。

**测试期覆盖**:
| 模型 | 测试期 | 天数 |
|------|--------|------|
| P0-clean/v2 | 03-04~03-10 | 7天 |
| P1 | 03-19~03-22 | 4天 |
| P2 | 03-13~03-22 | 10天 |
| P3 | 03-13~03-22 | 10天 |
| P4 | 03-13~03-22 | 10天 |
| P0-DA | 03-01~03-22 | 22天 |

**问题**: 测试期短且集中在3月。3月属于枯水季末/过渡期, 预测难度相对较低。**未覆盖汛期(6-10月)**, 而汛期电价行为完全不同(低价/零价/负价频发)。测试结果有偏。

### 4.2 时间权重策略

```python
# 半衰期 60天的指数衰减
w_time = exp(-ln(2) * days_ago / 60)

# 同水文季 ×2, 反季 ×0.5
w_season = 2.0 if same_season else 0.5

sample_weight = w_time * w_season
```

**评价**: 思路合理, 但存在**季节权重边界效应**:
- 3月15日测试: 训练数据中2月(枯季末)权重×2, 而4月(过渡期)权重×0.5
- 但4月电价行为可能比2月更接近3月 (都是枯→汛过渡)
- 硬性二分(枯/汛)不如使用连续月份相似度

**建议**: 使用水文季相似度评分替代硬0.5/2.0, 如 cosine_similarity(月份编码向量)。

### 4.3 P1中Similar-day筛选

P1实现的 find_similar_days 基于5个特征的欧几里得距离:
```python
features: avg_load, avg_renew, avg_hydro, avg_gap, avg_da
# 季节匹配: 距离×0.5 (同) / ×2.0 (异)
# 周末匹配: 距离×0.8 (同) / ×1.2 (异)
```

**评价**: 方法直观, 但存在以下问题:
1. 距离计算前归一化, 但各特征的分布可能不同, 同一标准化不稳健
2. 季节匹配的0.5/2.0系数差异太大, 可能导致只选同季节而忽略其他特征相似性
3. k=40的筛选后还加最近30天, 总训练样本可能差异不大
4. 用5个天级特征做相似度, 忽略了日内模式差异

## 七、关键缺陷分析

### 7.1 ✅ 已标记: v3.3 data leakage

P0-v3.3 (现已标记为含泄露) 的主要泄露来源:
1. 使用了D日的 `total_load` (实际负荷) → 当日结束时才可知
2. 使用了D日的 `renewable_output` 和 `hydro_output` → 同理

P0-clean和P0-model-v2已全部使用lag方式修复, 此问题已解决。

### 7.2 🚨 新发现: Savgol 分解泄漏 (P2)

详见 §3.4。Savgol滤波使用中心对称窗口, walk-forward时分解分量包含未来信息。

**影响**: moderate。特征重要性显示 trend/wave/spike 的贡献度约排在第15-25位, 不如价格lag。移除后MAE预计增加~2-5%。

### 7.3 🚨 新发现: 预测特征的时间对齐问题

P1和P2中使用了 `renew_fc`, `load_fc`, `gen_fc` 等预测特征。这些是D日发布的D+1预测, 理论上:
- 在walk-forward的test_date, 代码使用 `df['date_key'] == test_date` 来获取test_date当天的预测值
- 但预测值是针对 test_date 的(即 D日), 还是在 D-1 发布的?

需要检查: 代码中的 `renewable_forecast.forecast_date` 和 `date_key` 的映射。

从P1代码看:
```python
renew_fc_96 = pd.read_sql("""
    SELECT forecast_date as date_key, period, forecast_mw as renew_fc
    FROM renewable_forecast WHERE region='云南' AND category='总计'
""")
```
这里 `forecast_date` 被 alias 为 `date_key`, 而walk-forward中是 `df['date_key'] == test_date`。 
如果 `forecast_date` 表示"D日发布的预测(D+1日)", 那么 `forecast_date == test_date` 获取的是 D+1 的预测, 而不是 D 日的 → **这是对的, 因为预测是针对未来的**。

但需要确认: 在 D 日做预测时, 能看到 D 日的 forecast_date 吗? 通常 forecast 在 D-1 发布, 所以 `forecast_date = D-1` 发布供 D 日使用。

这个问题需要查看 renewable_forecast 表的实际数据来确认。如果 `forecast_date` 通过查错导致实际数据是 D 日的, 那就构成了泄漏。

### 7.4 💻 工程问题: Windows硬编码路径

## 八、具体改进建议

### 8.1 短期 (低风险, 高收益)

1. **移除BiLSTM**: 在5模型ensemble中, BiLSTM在所有测试日效果最差, 且增加大量复杂度。建议移除, 简化为4模型(GBR+LGB+XGB+Spread)。
2. **修复Savgol泄漏**: 改为滚动因果分解, 或直接移除分解特征。
3. **验证预测特征的时效性**: 检查renewable_forecast等表的 `forecast_date` 是否确实在D-1发布。
4. **合并P3和P4**: 如果P4相比P3的MAE改进<3%, 建议只保留更简洁的P3。

### 8.2 中期 (增加测试覆盖)

1. **扩展到汛期测试**: 将walk-forward测试期扩展到6-10月, 评估模型在零电价场景的表现。
2. **建立跨季节专用模型**: 枯期(spread模型最强)和汛期(可能需要分类模型处理零电价)分别建模。
3. **增加量化回归评估**: P2实现了P10/P90分位数, 但未在P3/P4中应用。建议统一量化评估框架。
4. **特征选择**: 对~80个特征做系统性的重要性排序和冗余分析, 将特征压缩到40-50个。

### 8.3 长期 (架构改进)

1. **统一特征工程**: 将dbt (PostgreSQL) 和Python (SQLite) 的特征工程统一到单一框架。
2. **配置驱动**: 所有超参数、DB路径、特征列表移到配置文件。
3. **MLflow全量接入**: 所有模型都做实验追踪, 便于版本对比。
4. **建立持续回测管道**: 自动化每日walk-forward回测, 取代手动运行脚本。

## 九、总结

### 项目优势
- 模型层次化设计清晰 (P0→P4)
- 特征工程覆盖全面 (价格/供需/电网/天气)
- walk-forward 回测设计正确
- 零泄露意识强 (P0-clean已修复v3.3的data leakage)
- DA和RT双任务覆盖

### 主要问题 (按严重程度排序)
1. **Savgol滤波时间泄漏** (P2分解, 影响~±5% MAE)
2. **BiLSTM无效增加复杂度** (10个测试日全部最差, 权重仅5%)
3. **特征膨胀/多重共线性** (~80个特征, 冗余度高)
4. **测试期集中在3月**, 汛期表现未知
5. **预测特征时间对齐** 需验证
6. **Windows硬编码路径**, 无法跨环境迁移
7. **双数据源分裂**, 特征工程重复

### 精度展望
- 3月测试期: MAE 35-45 (P3/P4), 目标50以下已经达成
- 全年平均: MAE 50-70 (预计, 含汛期和过渡期)
- **从70到50以下的改进路径**: 主要靠去除泄露(v3.3修复)+spread建模(P3)+汛期专用模型, 但由于每年电价特征不同, 50以下在全年尺度上仍有挑战
